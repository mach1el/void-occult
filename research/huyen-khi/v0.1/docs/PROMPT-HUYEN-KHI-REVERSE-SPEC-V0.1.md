# Implementation Prompt — Huyền Khí Reverse-Spec Research V0.1

Repository:

`st-mich43l/apexvoid-occult`

PR #90 has been merged into `master` as an experimental Annual Axes implementation.
Do not modify or recalibrate Annual Axes in this phase.

Create a new branch from the latest green `master`:

`research/ziwei-huyen-khi-reverse-spec-v0-1`

## Mission

Build a rigorous, provenance-first research system for a new **Huyền Khí**
engine.

This phase is not a production scoring implementation.

Its goals are:

1. collect and validate public numeric Huyền Khí outputs;
2. reproduce the corresponding natal chart facts with Calculation Core;
3. distinguish source-confirmed facts from output inferences and hypotheses;
4. create a controlled corpus and hypothesis-testing harness;
5. derive an explainable candidate reverse-spec without claiming access to the
   original proprietary formula;
6. prevent Huyền Khí from being confused with Xí Hoa, Đẩu Minh or Cung Khí.

Install the supplied data pack under:

```text
research/huyen-khi/v0.1/
```

Do not copy the pack into runtime knowledge directories yet.

# 1. Research ethics and source boundary

Use only publicly accessible pages.

Before any automated collection:

- inspect robots.txt and relevant site terms;
- use a descriptive user agent when permitted;
- enforce a maximum of six requests per minute;
- cache fetched pages;
- never bypass authentication, rate limits or anti-bot controls;
- never collect private account data;
- store only numeric facts and minimal metadata;
- do not copy long interpretive prose;
- provide a manual import mode when automated collection is not permitted.

Production runtime must never call or depend on the source site.

The source publisher states that its full theory is not public and warns that
imitations may be inaccurate. Therefore use the phrase:

```text
ApexVoid Huyền Khí Reverse-Spec
```

Do not label the result as the source site's exact formula unless unseen exact
reproduction is demonstrated and the repository owner explicitly approves the
claim.

18/18 displayed whole-chart totals

# 6. Controlled corpus design

Implement the supplied corpus plan.

## Phase A — manual gold corpus

Minimum 60 records:

- 20 positive totals;
- 20 negative totals;
- 20 totals near zero;
- all five Cục;
- both genders;
- all twelve hour branches;
- double-entry manual verification.

## Phase B — same-date 12-hour controls

Use public calendar pages that expose multiple hour charts for one date.

The same date controls year, month and day while hour changes:

```text
same date
× twelve hour branches
→ twelve chart structures and Huyền Khí outputs
```

Do not treat these as independent random samples in statistical uncertainty
calculations; preserve their group ID.

## Phase C — broad discovery corpus

Target 2,000 charts stratified by:

- Cục;
- Mệnh branch;
- major-star configuration;
- Vô Chính Diệu;
- Tuần/Triệt;
- natal Tứ Hóa targets;
- yin/yang and gender;
- hour branch.

Hold out complete birth-year or date groups so the model cannot memorize nearby
charts.

# 7. Feature ledger

Create a research feature catalog. Each feature must include:

```ts
interface HuyenKhiFeatureDefinition {
  featureId: string;
  description: string;
  sourceStatus:
    | "source-confirmed"
    | "output-inferred"
    | "experimental"
    | "unresolved";
  sourceIds: string[];
  extractionMethod: string;
  leakageRisk: "none" | "low" | "high";
}
```

Candidate feature groups:

1. Mệnh–Cục and Nạp Âm relations;
2. palace stem, branch and stem-branch Nạp Âm;
3. Cung Khí element mixture and Hỉ/Kị/Nhàn classification;
4. physical major stars and brightness;
5. physical minor stars and brightness;
6. Vô Chính Diệu;
7. Tuần and Triệt;
8. natal Tứ Hóa exact targets;
9. opposite, trine and adjacent configurations;
10. Thân location;
11. star combinations and transformation interactions.

Do not include:

- displayed Huyền Khí total as a feature for palace prediction;
- displayed palace score from a related page;
- source-page classification text that directly encodes the target;
- Annual Axes, Palace Overview, Major Fortune or Monthly Flow scores.

# 8. Score-alphabet analysis

The seed corpus shows many quarter-step values plus smaller residual values.

Build reports for:

- unique palace-score values;
- frequency by 0.25-grid membership;
- residual modulo 0.25;
- value range by palace name;
- value range by Cục;
- value range by major-star configuration;
- count of exact integers, halves, quarters and residual decimals.

This supports—but does not prove—the hypothesis:

```text
coarse base score + fine modifiers
```

Do not encode that hypothesis as production truth before matched-pair tests.

---

# 9. Matched-pair research

Build a matcher that finds pairs differing in as few facts as possible.

Priority experiments:

1. same Mệnh branch and Cục, different main-star brightness;
2. same main-star configuration, Tuần/Triệt differs;
3. same palace stars, one natal Tứ Hóa target differs;
4. same date, adjacent hour branches;
5. Vô Chính Diệu with and without strong opposite palace;
6. same star in focus, opposite and trine positions;
7. same coarse quarter score with different residual decimals.

For each pair report:

```text
facts shared
facts changed
observed palace-score delta
candidate rule explanations
confounders
```

Do not call a coefficient established from one pair.

# 10. Candidate model ladder

Build models in this order.

## Model 0 — null baseline

Predict palace-name mean from the training set.

## Model 1 — coarse lookup baseline

Use only:

- palace name;
- Cục;
- Mệnh branch;
- major-star configuration;
- brightness.

## Model 2 — explainable additive rules

Use versioned atomic rules:

```ts
palaceScore = sum(appliedRuleDeltas);
```

Every delta must have a rule ID and feature provenance.

## Model 3 — bounded interactions

Add only interactions supported by matched pairs or stable repeated residuals:

- Tuần/Triệt × star state;
- Tứ Hóa × exact target;
- VCD × opposite major-star structure;
- Mệnh–Cục relation × palace structure.

## Model 4 — symbolic hypothesis search

Symbolic regression, trees or sparse regression may propose candidate rules.
They are research tools only.

A black-box model may be used to measure whether the feature set contains enough
signal, but it cannot become the production Huyền Khí engine.

Do not use neural networks or unconstrained high-cardinality memorization as the
first approach.
